# システム改善計画書 (improvement.md)

## 優先度：高（効果が大きい／記事が増えると顕在化）

### 1. DB インデックスの不足【済】
`models.py` の `Post` には `created_at`・`genre`・`is_published`・`user_id` にインデックスがありません。トップの `ORDER BY created_at DESC`、ジャンル絞り込み、公開状態フィルタ、マイページの `user_id` 絞り込みはすべて全表スキャンになります。記事数が増えると一覧・検索・詳細ページが目に見えて遅くなります。
* **対処方法:** `db.Column(..., index=True)` を付与し、マイグレーションを追加。特に `created_at`・`is_published`・`genre`。複合インデックス `(is_published, created_at)` も有効です。

### 2. アップロード画像が未最適化のまま配信されている【済】
`_save_images()` は原本を UUID 名で保存するだけで、リサイズや圧縮を行っていません。一覧のサムネイルは CSS 上 260×158px なのに、数 MB の原寸画像がそのまま読み込まれます。読み込み速度に最も直結する部分です。
* **対処方法:** アップロード時に Pillow でサムネイル用の縮小版（例：幅 400px の WebP）を生成し、`thumbnail_img` はそれを指すようにする。本文画像も長辺を上限（例：1600px）で縮小。

### 3. マイページが全投稿を一括ロードしている【修正中】
`mypage()` は `Post.query...all()` で全件取得し、テンプレートで全カードを描画してから JS で隠す方式です。index はサーバーページネーション化済みなのに、ここだけ古い方式が残っています。投稿が増えるとメモリ・DOM が肥大します。
* **対処方法:** index と同様に `paginate()` に統一。`selectinload(Post.hashtags)` も付与。

## 優先度：中

### 4. ページ固有 CSS を `<body>` 内で読み込んでいる
`index.html` などは `{% block content %}` の中（＝body 内）で `<link rel="stylesheet">` を読み込んでいます。レンダリング途中で読み込むためチラつき（FOUC）や再レイアウトの原因になります。
* **対処方法:** `base.html` に CSS ブロックを設けて `<head>` 側で読み込む、または preload。

### 5. Markdown を毎回パースしている
`detail()` は本文（Markdown ＋独自タグ）をアクセスのたびに変換しています。本文は編集時しか変わらないので無駄が大きい。
* **対処方法:** レンダリング結果を列（例：`body_html`）に保存して投稿・編集時にだけ生成、詳細表示はそれを出力。もしくはメモリキャッシュ。

### 6. `User.query.get()` が非推奨 API
`app.py` の `load_user` の `User.query.get()` は SQLAlchemy 2.0 で Legacy 扱いです。
* **対処方法:** `db.session.get(User, int(user_id))` に置き換え。

### 7. キーワード検索が前方ワイルドカードで全走査
`Post.title.ilike('%word%')` は先頭 `%` のためインデックスが効かず全表スキャンです。小規模なら問題ありませんが規模拡大時のボトルネックになります。
* **対処方法:** 将来的には PostgreSQL の全文検索（`pg_trgm`／`tsvector`）を検討。当面は index 化と併用。

### 8. `<img>` に width/height がなくレイアウトシフトが起きる
サムネイル等に intrinsic なサイズ指定がないため、読み込み時に CLS（表示のガタつき）が発生します。
* **対処方法:** `_macros.html` などの `<img>` に `width`／`height` を明示。

### 9. ログインのユーザー名入力が `type="password"`
`login.html` でユーザー名欄が `type="password"` になっており、`autocomplete="username"` と不整合です（意図的な目隠しなら許容ですが、ブラウザ警告や自動入力の不具合の原因になります）。
* **対処方法:** 意図がなければ `type="text"` に。目隠しが目的なら意図をコメントで明記。

### 10. 認証まわりの軽微なハードニング
ゲートキー／Cookie の比較が単純な `==`（非定数時間）で、ブルートフォース対策もセッション単位のため別ブラウザで回避可能です（コード内でも指摘済み）。
* **対処方法:** `secrets.compare_digest()` での比較、IP ベースの制限（Flask-Limiter）。

## 優先度：低（規模が小さいうちは実害小）

### 11. 静的アセットのバンドル／バージョニングがない
CSS が毎ページ複数ファイルに分かれ、キャッシュバスティングもなし。更新時にキャッシュが残る懸念。
* **対処方法:** クエリ文字列やハッシュ付きファイル名で対応。

### 12. `<title>` が全ページ固定
`base.html` が `Mycreate1.MITOblog.jp` 固定で、記事タイトルが反映されず SEO・共有時に不利。meta description もなし。
* **対処方法:** per-page `<title>` ブロック化。

### 13. ラベルの `for=""` が空・不一致
`login.html` などで label とフォーム要素が関連付いておらずアクセシビリティ上の欠陥。
* **対処方法:** `for` と `id` を対応。

### 14. `sync_hashtags` の同名タグ同時作成レース
「検索 → なければ作成」の間の競合。単一管理者運用なら実害は低い。
* **対処方法:** commit 時の `IntegrityError` を捕捉して再取得。

### 15. 関連記事の 4 連続クエリ・ジャンル一覧の重複クエリ
`_get_related_posts` は最大 4 回、index はジャンル一覧を毎回問い合わせ。
* **対処方法:** まとめ or 軽いキャッシュ。

### 16. 本文中 Markdown 画像に `loading="lazy"` がない
記事下部の画像も初期ロードされる。
* **対処方法:** 遅延読み込み付与。

### 17. DiceBear アバターへの外部依存
ページごとに外部リクエスト。
* **対処方法:** ローカル生成・キャッシュを検討。
